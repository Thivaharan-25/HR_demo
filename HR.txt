Handles all aspects of user identity, access control, and first-time setup. This module is the entry point for all roles and governs security posture for the entire platform.

──────────────────────────────────────────────────────────────────────────────
2.1 Login Methods
──────────────────────────────────────────────────────────────────────────────

2.1.1 Email & Password
How it functions:
- Employee enters registered email and password on the login screen.
- System validates against the organisation's user directory (bcrypt-hashed passwords).
- On success: access token (JWT, 8-hour expiry) and refresh token (30-day rolling) issued; session created.
- On first failed attempt: error shown. After 3 failed attempts: 15-minute lockout countdown displayed. After 5 failed attempts: account locked; HR Admin receives alert; employee must use 'Unlock Account' flow via email.
- Password strength enforced: minimum 8 characters, 1 uppercase, 1 number, 1 special character. Passwords expire every 90 days (configurable per org).

2.1.2 Single Sign-On (SSO)
How it functions:
- Supported protocols: SAML 2.0, OAuth 2.0 / OpenID Connect. Compatible providers: Google Workspace, Microsoft Azure AD, Okta, custom SAML IdPs.
- Employee taps 'Sign in with [Provider]' → redirected to provider's authentication page → on success, SAML assertion or OAuth token returned to platform.
- Platform validates token signature, maps provider user attributes (email, name, department) to platform user record.
- If no matching platform account found: access denied; HR Admin notified to provision the account.
- SSO sessions honour provider session duration; platform refresh token tracks local session independently.

2.1.3 Biometric Authentication — Two Distinct Systems
Revised v3.0 | March 2026 — Supersedes previous biometric spec.
The platform uses two completely separate biometric systems. They serve different purposes, operate on different hardware, store data differently, and must never share templates or identity tokens.

  System A — Mobile Biometric | Purpose: App login + Remote/Hybrid clock-in | Biometric lives: On device only (Secure Enclave / Keystore) | Managed by: Employee
  System B — Hardware Device  | Purpose: Onsite clock-in / clock-out         | Biometric lives: On physical terminal only                   | Managed by: HR Admin

⚠  GDPR / PDPA NOTE (Critical for Production): Biometric data is Special Category personal data under GDPR Art. 9. Before any enrolment, the platform MUST obtain explicit, separate, written consent from the employee. Consent must be: specific to purpose, freely given (non-coerced), revocable at any time, and stored immutably with timestamp and version. Withdrawal of consent must trigger immediate template deletion from all devices.

──────────────────────────────────────────────────────────────────────────────
SYSTEM A — Mobile Biometric (App Login & Remote/Hybrid Clock-In)
──────────────────────────────────────────────────────────────────────────────

A.1 Purpose & Scope
Used exclusively for authenticating an employee into the mobile app (replaces password entry after first setup) and for verifying identity when clocking in/out under Remote or Hybrid modes. Not used for onsite clock-in/out — that is System B only.

A.2 Architecture — Challenge-Response + Asymmetric Key Pair
The biometric template never leaves the device. The server never sees, stores, or processes any biometric data.
Core trust model: The device holds a private key locked behind the biometric sensor. The server holds the matching public key. When the employee passes the biometric check, the device uses the private key to sign a server-issued nonce. The server verifies the signature. No biometric data is transmitted at any point.

First Enrolment Flow:
1. Employee completes first login via email/password.
2. App prompts 'Enable Biometric Login?' → employee taps Enable → OS biometric prompt shown.
3. On biometric success, device generates asymmetric key pair (RSA-2048 or EC P-256):
   - Private key stored in Secure Enclave (iOS) or Android Keystore — hardware-backed, non-exportable.
   - Private key access requires biometric authentication on every use (no passthrough).
4. Device sends ONLY the public key to the platform, along with: { device_id, device_model, os_version, app_version, enrollment_timestamp }.
5. Server stores: { employee_id, device_id, public_key, enrolled_at, status: active } — the Device Credential Record.

Login / Clock-In Flow:
1. App requests a challenge nonce from server: GET /auth/biometric/challenge → server returns { nonce, expires_at (30 seconds) }. Nonce is single-use.
2. App triggers OS biometric prompt.
3. On success, device signs: signature = sign(privateKey, nonce + timestamp + device_id).
4. App sends: { employee_id, device_id, nonce, signature, timestamp }.
5. Server verifies: nonce not expired, not used before, signature valid against stored public key, device status active, timestamp within ±60 seconds.
6. On success: nonce consumed → JWT issued → clock-in record created.

A.3 Multi-Device & Device Replacement
- Employee can register multiple devices; each gets its own key pair and Device Credential Record.
- When phone is replaced: old private key is hardware-locked and cannot be transferred (by design). Employee re-enrols on new device after password login. Old device record set to status: inactive.
- If phone is lost: HR Admin revokes old device record → all signatures from that device_id are rejected.

A.4 Failure Handling
- 1-2 consecutive failures: retry prompt.
- 3 failures in one session: fallback to password/SSO for that session. Biometric not disabled.
- 5 failures across sessions: biometric disabled on device. Employee re-enrols via Settings after password login.
- Nonce expired (>30s): request new challenge. Never retry with old nonce.
- Device clock out of sync (>60s drift): clock-in rejected with 'Device time mismatch' error.

A.5 Mobile Biometric Consent
Before enrolment prompt is shown, app displays a Biometric Consent Screen explaining: what is stored (key pair only — no biometric image), where (on device only), what it is used for, and how to revoke. Employee must tick consent checkbox and confirm. Consent stored: { employee_id, device_id, consent_type: 'mobile_biometric', consented_at, version }. Revoking consent deletes the Device Credential Record and disables private key access.

──────────────────────────────────────────────────────────────────────────────
SYSTEM B — Hardware Biometric Device (Onsite Clock-In / Clock-Out)
──────────────────────────────────────────────────────────────────────────────

B.1 Purpose & Scope
Physical biometric terminals at office entrances, factory gates, or onsite entry/exit points. These are the sole mechanism for recording onsite attendance. Employees do not use the mobile app to clock in when physically at the office.

B.2 Supported Hardware
Device Types: Fingerprint scanner optical (ANSI/ISO 19794-2), Fingerprint scanner capacitive (ANSI/ISO 19794-2), Face recognition terminal IR-based with liveness detection (ISO/IEC 19794-5), Finger-vein / palm-vein reader (ISO/IEC 19794-6), Multi-modal fingerprint + face.
Supported Manufacturer SDKs (out-of-box): ZKTeco (ZKLib SDK), Suprema (BioStar 2 API), HikVision (ISAPI), Anviz (CloudClocking API), Generic REST adapter for any manufacturer supporting REST push.

B.3 Core Data Principle — What Is Stored Where
Biometric template (actual fingerprint/face data): stored on device flash memory ONLY — NEVER on platform server.
Employee token (UUID mapped to employee_id): stored on device alongside template AND on platform server (maps token → employee_id).
Match result / match score: generated on device, received by platform via event payload.
Clock event (in/out, timestamp, device_id): queued locally then persisted as attendance record.
Raw biometric image / scan: NEVER retained after match — NEVER received by server.
If the server is breached, no biometric data is exposed. The device holds templates; the server holds only tokens and events.

B.4 Device Registration Flow
1. HR Admin → System Configuration → Hardware Devices → 'Register New Device'.
2. Form fields: Device Name, Device Type, Manufacturer & Model, Serial Number, Branch/Location (determines geo-zone), LAN IP Address (static), API Key / SDK Credentials, Communication Protocol (REST Push / MQTT / SDK).
3. Platform generates a Device Secret (HMAC-SHA256 key) unique to this device. This secret is entered into the device configuration so it can sign outbound event payloads.
4. Platform sends test ping to device IP. On success → device status: Online.
⚠  All communication between device and platform must be over TLS 1.2+. Each event payload must include an HMAC-SHA256 signature computed using the Device Secret, preventing spoofed events.

B.5 Employee Biometric Consent — Hardware Device
Before any employee is enrolled on a hardware device, the platform checks for an active Hardware Biometric Consent record. If not present, platform sends a Consent Request notification (in-app + email) to the employee. Employee reviews: what biometric is captured, where it is stored (physical device only, never servers), what it is used for (onsite attendance only), right to withdraw. Employee taps 'I Consent' → consent stored immutably. If employee does not consent, HR Admin is notified and an alternative attendance method must be configured.

B.6 Employee Enrolment Flow
1. After consent is obtained, HR Admin → Employees → [Employee] → 'Enrol on Device' → select target device(s).
2. Platform creates Enrolment Session: { employee_id, device_id, session_token, expires_at (24 hours), status: pending }.
3. Platform pushes session to device. Device displays employee name and ID, ready to capture.
4. HR Admin or Site Admin calls employee to terminal. Employee presents biometric (3 scans averaged for fingerprint).
5. Device creates biometric template stored internally, linked to a Platform Token (UUID generated by platform — sent as part of enrolment session). The UUID is the only identifier on the device that links to a real employee.
6. Device sends enrolment confirmation: { session_token, employee_token (UUID), device_id, template_quality_score, enrolled_at }.
7. Platform stores Device Enrolment Record: { employee_id, device_id, employee_token, enrolled_at, status: active }.
Multi-device enrolment: each device gets its own employee_token UUID — same employee has different tokens on different devices. If a device is stolen, the attacker sees UUIDs only.

B.7 Onsite Clock-In / Clock-Out Flow (End to End)

Clock-In on Device:
1. Employee presents biometric at terminal.
2. Device performs 1:N local match against all enrolled templates in internal memory. No network required for the match itself.
3. On successful match, device retrieves employee_token linked to matched template.
4. Device builds signed event payload:
   { event_id (UUID), device_id, employee_token, event_type: 'clock_in', timestamp_utc, match_score, match_method, liveness_check: 'passed', hmac_signature }
5. Device sends payload to platform via configured protocol (REST POST / MQTT).
6. Device shows success indicator to employee (green light / beep).

Platform Processing (server-side):
1. Validate HMAC signature against Device Secret → reject if invalid.
2. Check event_id uniqueness → reject if already processed (prevents duplicate records).
3. Check timestamp_utc is within ±5 minutes of server time → reject if outside window (prevents replay attacks).
4. Look up employee_token in Device Enrolment Records → resolve to employee_id.
5. Verify employee is Active (not terminated).
6. Check for existing open clock-in for today → if found, log Duplicate Clock-In Detected, notify HR, do not create second record.
7. Create Attendance Record: { employee_id, date, mode: 'Onsite', source: 'hardware_biometric', device_id, device_name, location, clock_in_utc, clock_out_utc: null, match_score, match_method, liveness_check, status: 'open', flags: [] }.
8. Apply Late Arrival check: if clock_in > shift_start + grace_period → add LATE_ARRIVAL flag.
9. Push real-time notification to employee mobile app.

Clock-Out: same flow with event_type: 'clock_out'. Platform finds open attendance record, updates clock_out_utc, calculates total_work_hours. Applies SHORT_HOURS and EARLY_DEPARTURE checks.

B.8 Event Delivery & Reliability
Device-Side Queuing: if device cannot reach platform, events written to device's internal persistent queue (flash storage, not RAM). Retry: 30s, exponential backoff to 30 min max. Queue holds up to 10,000 events.
Platform Processing on Reconnect: events processed in chronological order. Timestamp within ±4 hours → processed normally, marked source_sync: realtime. Gap > 4 hours → flagged source_sync: late_sync → HR Admin review required. Duplicate event_id values silently deduplicated.
Heartbeat: device sends heartbeat every 5 minutes (configurable): { device_id, timestamp, queue_depth, firmware_version, storage_free_pct }. No heartbeat for 10 min → status: Warning. No heartbeat for 30 min → status: Offline, HR Admin alert.

B.9 Security Controls
- Transport: TLS 1.2+ for all device ↔ platform communication.
- Event authenticity: HMAC-SHA256 on every payload using per-device secret.
- Anti-replay: event_id deduplication + timestamp window validation (±5 min).
- Token anonymisation: employee_token is a UUID with no PII — device cannot identify employee from token alone.
- Template isolation: biometric templates stored only on device, never transmitted.
- Device secret rotation: HR Admin can rotate device's HMAC secret from admin panel; old secret invalidated immediately.

B.10 Employee Termination & Template Deletion
When employee is terminated or consent withdrawn:
1. Platform immediately sets all Device Enrolment Records for employee to status: revoked.
2. Platform sends Template Deletion Command to each enrolled device: { command: 'delete_template', employee_token, issued_at, hmac }.
3. Device deletes template linked to employee_token from internal storage.
4. Device sends Deletion Confirmation: { employee_token, deleted_at, device_id }.
5. Platform records deletion confirmation — required for GDPR compliance (right to erasure).
6. If device is offline when deletion commanded: command queued. Delivered on reconnect before any new events processed.
⚠  Production Requirement: Template deletion confirmation must be obtained within 72 hours of termination (GDPR-aligned). If device is offline beyond this window, HR Admin receives escalating alert.

B.11 Failure & Fallback Handling
- 1-2 match failures: retry prompt.
- 3 consecutive failures: device offers PIN Entry fallback (if HR Admin has enabled PIN fallback for this device). PIN event logged as source: pin_override_biometric_failure.
- Liveness check fails (face terminal): match rejected. If same employee triggers 3+ liveness failures in a day → HR Admin alerted (potential spoofing attempt).
- Device hardware failure: HR Admin receives 'Device Offline' alert. HR Admin can manually log attendance for affected period via Attendance Corrections flow.
- Device storage full: HR Admin alerted. Oldest already-synced events purged first. Unsynced events never auto-purged.

B.12 HR Admin — Device Management Panel
Device Registry View: Device Name, Location, Type, Manufacturer, Status (Online/Warning/Offline/Error), Last Heartbeat timestamp, Today's Event Count, Enrolled Employees Count, Pending Deletion Commands.
Per-Device Actions: Test Ping, View Event Log (filterable: date, employee, event type, match result, source), View Enrolled Employees (with enrolment date, status, template quality score), Rotate HMAC Secret, Configure (heartbeat interval, PIN fallback, liveness sensitivity, auto-clock-out window), Decommission Device (triggers bulk template deletion for all enrolled employees).
Enrolment Management: Enrol employee on device, Re-enrol (if template quality degraded), Remove enrolment (triggers template deletion command), Transfer enrolment to replacement device.

DEMO vs PRODUCTION NOTES
Mobile biometric: Demo simulates with mock challenge-response. Production: real FIDO2/WebAuthn or platform biometric API (Face ID / Touch ID / Android Biometric).
Hardware device events: Demo uses 'Simulate Scan' button in admin panel. Production: real device pushes events via REST/MQTT with HMAC authentication.
HMAC validation: Skipped in demo. Required in production — every inbound event validated against per-device HMAC secret.
Template deletion: Demo removes enrolment record. Production: real deletion command sent to device firmware; confirmation required within 72 hours.
Heartbeat monitoring: Demo uses 30-second polling. Production: real-time WebSocket or MQTT subscription from device.

2.1.4 Multi-Factor Authentication (MFA)
How it functions:
- MFA is optional for Employees and mandatory for HR Admin and Organisation Owner roles.
- Supported methods: TOTP (Google Authenticator, Authy), SMS OTP (fallback only), Email OTP (fallback only).
- TOTP setup: QR code generated at enrolment; employee scans with authenticator app; must verify with first code to confirm setup.
- On login with MFA enabled: after password/SSO step, 6-digit TOTP or OTP prompt appears. Code valid for 30 seconds (TOTP) or 10 minutes (email/SMS).
- Backup codes: 8 single-use codes generated at MFA setup. Employee can download/print; HR Admin can regenerate if all used.

──────────────────────────────────────────────────────────────────────────────
4.1 Onsite Attendance — Hardware Biometric Device
──────────────────────────────────────────────────────────────────────────────

Overview:
Onsite attendance is recorded exclusively through the hardware biometric terminals installed at the organisation's entry/exit points. Employees do not use the mobile app to clock in or out when they are physically at the office. The platform receives event data from the device and creates attendance records automatically. See Section 2.1.3B for full device integration architecture.

Clock-In Flow:
How it functions:
- Employee approaches the biometric terminal at the office entrance.
- Employee places finger on the fingerprint scanner or stands in front of the face recognition camera.
- Device performs a local 1:N biometric match against enrolled templates stored on the device (no internet required for the scan itself).
- On successful match: device transmits an event to the platform — { device_id, employee_token, event_type: 'clock_in', timestamp_utc, match_score, match_method }.
- Platform processes the event:
  - Looks up employee by device token → confirms employee is Active and enrolled on this device.
  - Validates device's registered geo-zone (device's location is pre-configured; geo-zone attached to device registration — see Section 2.1.3B).
  - Creates attendance record: mode = 'Onsite', source = 'hardware_biometric', device_id, timestamp, match_score.
  - No mobile GPS or IP check needed — the physical device's location is the attendance proof.
- Employee's mobile app updates to show 'Clocked In (Onsite – Office Terminal)' within seconds of the event being received. Employee receives in-app push notification confirming the clock-in.
- If employee does not have the app open: attendance is still recorded. The device operates independently of the mobile app.

Clock-Out Flow:
- Employee scans at the terminal on departure. Device sends clock_out event.
- Platform calculates: total_work_hours = clock_out_timestamp − clock_in_timestamp, minus tracked breaks if applicable.
- Shift compliance check: if total hours < required shift hours − threshold (configurable, default 30 min), record flagged as 'Short Hours' for HR review.
- If employee forgets to scan out: auto-clock-out applied at org-configured cutoff (default 23:59). Auto-clock-out creates a correction request flagged to HR. Employee notified via push notification.

Break Tracking (Optional, configurable per org):
- 'Start Break' button available during active clock-in. Creates a break segment with timestamp.
- 'End Break' button resumes work timer. Total break time displayed and deducted from work hours.
- Org can configure: required minimum break (e.g., 30 min if shift > 6 hours); system alerts if break not taken.
- Maximum break cap configurable (e.g., 2 hours); breaks exceeding cap flagged.

Late Arrival & Early Departure Logic:
- Late Arrival: if clock-in time > shift_start_time + grace_period (default 15 min, configurable), record flagged as 'Late'. Late count tracked; HR can configure thresholds for automatic escalation or leave deduction policy.
- Early Departure: if clock-out time < shift_end_time − threshold (default 30 min), flagged as 'Early Departure'. Employee prompted to add reason note.

──────────────────────────────────────────────────────────────────────────────
4.2 Remote Attendance
──────────────────────────────────────────────────────────────────────────────

Overview:
Employee works from a non-office location (home, client site, co-working space). No geo-fence required, but VPN and device validation apply based on org policy.

How it functions:
- Employee taps 'Clock In' → selects 'Remote' → optionally enters work location note (e.g., 'Home – Colombo', 'Client site – London').
- System captures: device IP, device fingerprint, timestamp.
- Device check: if device ID not in employee's registered devices list, additional OTP verification prompted before clock-in proceeds.
- Clock-out flow same as onsite minus geo-fence check.

Remote-Specific Features:
- Daily Remote Limit: org can set max consecutive remote days (e.g., 5 days) before manager approval required for further remote days. Threshold triggers a 'Remote Extension Request' flow.
- Remote Work Reason: configurable as optional or mandatory per org. Dropdown options: Home Office, Client Site, Travel, Medical, Other.
- Timezone Support: employee can set a working timezone (within allowed range set by HR). Timestamps stored in UTC; displayed in employee's set timezone. Shift compliance calculated relative to org's timezone.

──────────────────────────────────────────────────────────────────────────────
4.3 Hybrid Attendance
──────────────────────────────────────────────────────────────────────────────

Overview:
Employee works from multiple locations in a single day — for example, morning at the office and afternoon from home. System records each segment independently and calculates a composite daily summary.

How it functions:
- Employee clocks in under any mode (Onsite or Remote) to begin the first segment.
- During the active shift, employee taps 'Switch Mode' → selects new mode → system ends current segment (records segment end timestamp and duration) and immediately opens a new clock-in for the new mode.
- Each segment independently validated by its respective rules:
  - Onsite segment: recorded via hardware biometric terminal at the office. Employee scans in/out on the physical device. Platform receives device event and creates the onsite segment record.
  - Remote segment: employee uses the mobile app to clock in/out; mobile GPS + device fingerprint captured.
- Each mode switch creates a new segment record: { mode, start, end, duration, validation_flags }.
- Daily attendance record aggregates all segments: total_onsite_hours + total_remote_hours = total_work_hours.
- Daily attendance card shows hybrid breakdown: 'Onsite 4h 30m | Remote 3h 15m | Total 7h 45m'.

Hybrid-Specific Rules:
- Maximum mode switches per day: configurable (default 3). Prevents abuse. If max reached, additional Switch Mode is blocked; employee prompted to contact HR.
- Segment gap handling: if > configurable threshold (default 30 min) elapses between clock-out of one segment and clock-in of next, gap is logged as 'Unrecorded Gap'. Employee prompted to explain gap (dropdown + note). Gap can be marked as break, travel, or personal by HR.
- Hybrid records appear distinctly in attendance reports, with full segment-level breakdowns available for audit.

──────────────────────────────────────────────────────────────────────────────
4.4 Offline Mode & Background Sync
──────────────────────────────────────────────────────────────────────────────

How it functions:
- Employee taps Clock In / Clock Out with no internet connectivity. App detects offline state.
- Action queued in on-device SQLite queue: records { action_type, mode, gps_snapshot, device_time, device_id, network_state }.
- UI displays persistent banner: 'Attendance queued – will sync when online.' Vibration feedback confirms action captured.
- GPS snapshot taken at moment of queue (device may cache last known location if GPS unavailable indoors; accuracy recorded).
- On internet reconnection: background sync sends queued items in chronological order.

Server validation for queued items:
- Timestamp within acceptable sync window (default ±4 hours from device time): accepted and processed normally.
- Timestamp outside window: flagged as 'Late Sync – Requires Review'. HR Admin notified; record created with 'Pending Verification' status.
- Duplicate record detected (server already has entry for that employee/day/mode): conflict logged; HR must reconcile.
- Employee receives push notification confirming sync success or sync failure with instructions.

- Corrections audit trail: every change logged with: who changed, when, from→to values, reason. Immutable.

5. LEAVE MANAGEMENT MODULE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Handles the complete leave lifecycle: entitlement configuration, request submission, approval workflows, balance tracking, and calendar visibility. Supports multiple leave types with org-configurable rules.

──────────────────────────────────────────────────────────────────────────────
5.1 Leave Types & Entitlement
──────────────────────────────────────────────────────────────────────────────

Supported Leave Categories:
- Annual Leave (AL) – Standard paid leave accrued based on employment tenure.
- Medical / Sick Leave (ML) – Paid leave for illness; may require medical certificate above configured threshold (e.g., > 2 consecutive days).
- Emergency / Compassionate Leave – Short-notice leave for personal emergencies; limited entitlement per year.
- Maternity / Paternity Leave – Statutory leave; duration governed by jurisdiction (UK: statutory maternity/paternity rules; Sri Lanka: Labour Act provisions auto-applied based on org country setting).
- Unpaid Leave (UL) – Leave without pay; requires manager + HR approval.
- Public Holidays – Automatically populated from org country/region calendar. Employees do not submit requests for these.
- Compensatory Off (Comp Off) – Earned for working on public holidays or weekends. Configurable per org.
- Custom Leave Types – HR Admin can create up to 10 custom leave types per org (e.g., Study Leave, Volunteer Day).

Entitlement & Accrual Logic:
- Entitlement configured per leave type per employment band/grade (HR Admin sets in config).
- Accrual options: upfront (full entitlement on Jan 1 or employment anniversary), monthly accrual (entitlement ÷ 12 credited monthly), or tenure-based (increases at 1, 3, 5 years — configurable thresholds).
- Pro-rating: employees who join mid-year receive pro-rated entitlement based on remaining months. Calculation: (annual_entitlement ÷ 12) × remaining_months. Configurable to round up or down.
- Carry-forward rules: configurable per leave type — max carry-forward days, expiry date for carried days (e.g., must use by 31 March).

──────────────────────────────────────────────────────────────────────────────
5.2 Leave Request Submission
──────────────────────────────────────────────────────────────────────────────

How it functions:
- Employee navigates to Leave → taps 'Apply Leave'.
- Form fields:
  - Leave Type (dropdown; only types applicable to employee's band shown).
  - Start Date / End Date (date picker; past dates selectable only for retrospective leave, configurable).
  - Duration: system auto-calculates working days between start and end, excluding weekends and public holidays. Employee can select 'Half Day' (AM/PM) if org enables half-day leave.
  - Balance preview: 'After this leave: X days remaining' shown dynamically as dates are selected.
  - Reason (text field; configurable as mandatory or optional per leave type).
  - Attachment (optional for standard AL; mandatory for Medical Leave if > configured threshold days).

Conflict checks run on submission:
- Overlapping with existing approved/pending leave → error: 'Conflict with existing leave on [date].'
- Insufficient balance → error with exact shortfall. If org allows negative balance (configurable), a warning shown but submission proceeds with HR flag.
- Blackout periods (e.g., payroll close week, peak season — configurable by HR) → warning shown; submission blocked or flagged depending on org setting.
- Team leave limit: if org sets 'max N employees on leave simultaneously per department', check runs → if limit reached, employee shown a warning and request queued for HR review.
- On successful submission: request enters 'Pending' state. Notification sent to approver (team lead if employee has one, else HR Admin).
- Auto-approval: HR Admin can configure certain leave types (e.g., single-day AL with sufficient balance) to auto-approve without human review.

──────────────────────────────────────────────────────────────────────────────
5.3 Approval Workflow
──────────────────────────────────────────────────────────────────────────────

How it functions:
- Pending request appears in approver's Pending Approvals inbox (notification badge on dashboard).
- Approver (team lead / HR Admin) views request detail: leave type, dates, duration, balance impact, reason, attachment.
- Team calendar view alongside request: shows other team members on leave for same period (helps judge coverage).

Actions:
- Approve: leave status → 'Approved'. Balance deducted. Employee notified. Attendance auto-marked as 'On Leave' for approved dates.
- Reject: mandatory reason field (dropdown + note). Status → 'Rejected'. Employee notified with reason.
- Request More Info: sends a query back to employee; request stays 'Pending' until employee responds or withdraws.
- Escalation: if approver does not action within configured SLA (default 48 hours), reminder notification sent. If still unactioned after 72 hours, escalated to HR Admin automatically.
- Multi-level approval (optional, configurable per org): e.g., team lead approves first, then HR Admin for leave > 5 days. Each level tracked with timestamp.

──────────────────────────────────────────────────────────────────────────────
5.4 Leave Calendar
──────────────────────────────────────────────────────────────────────────────

- Monthly/weekly calendar view showing employee's own leave (coloured by leave type).
- Team calendar (team leads): overlaid view of all direct reports' approved leave. Colour-coded by employee. Toggle to show/hide individual team members.
- 'Team Coverage' indicator per day: shows % of team present. Days below org-configured minimum coverage threshold highlighted in amber/red.
- Public holidays marked automatically based on org's country/region setting.

──────────────────────────────────────────────────────────────────────────────
5.5 Leave Balance Dashboard
──────────────────────────────────────────────────────────────────────────────

- Tabular view of all leave types: entitlement, used, pending, remaining, carry-forward from prior year (if applicable).
- Visual bar for each type showing utilisation %.
- Transaction history: chronological log of all leave events (accrual credits, deductions, adjustments with reason and who made them).

──────────────────────────────────────────────────────────────────────────────
5.6 HR Admin Leave Management
──────────────────────────────────────────────────────────────────────────────

- Manual balance adjustments: HR Admin can add/deduct days from any employee's balance with mandatory reason (audit-logged).
- Bulk entitlement update: adjust entitlements for an entire grade/band or department at once.
- Leave policy configuration: all accrual rules, carry-forward caps, blackout periods, approval tiers, auto-approval rules managed here.
- Leave utilisation report: org-wide, filterable by department, leave type, date range.

- Expired certifications: status changes to 'Expired'. Linked skill validation reverts to 'Unvalidated' if cert was the only validation source. Employee prompted to renew.

- Each gap entry shows: skill name, required proficiency, current proficiency (or 'Not Present'), gap severity (High / Medium / Low based on whether it's Core or Preferred and the proficiency delta).
- 'Start Learning' button per gap: links to LMS course or suggested development goal (if LMS integration active).
- Report refreshed in real-time as skills are added/validated.

8. TALENT DEVELOPMENT MODULE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Supports structured employee growth through development plans, OKRs, and goal tracking. Plans can be created individually or in bulk by HR Admin and linked to organisational objectives.

──────────────────────────────────────────────────────────────────────────────
8.1 Development Plans
──────────────────────────────────────────────────────────────────────────────

8.1.1 Viewing Active Plans (Employee):
- Employee sees all active development plans assigned to them, sorted by due date.
- Each plan shows: plan name, overall progress % (computed from goal completions), start date, target end date, assigned by (HR Admin / manager), plan status (Active / Completed / Overdue / Paused).

8.1.2 Development Goals:
Goal types supported:
- Skill Goal: acquire or improve a specific skill to a target proficiency level. Completion auto-detected when skill validation meets target level.

──────────────────────────────────────────────────────────────────────────────
8.2 OKRs & KPIs
──────────────────────────────────────────────────────────────────────────────

How it functions:

- HR Admin cascades objectives to departments and individuals. Employees see their assigned OKRs in the Talent Development module.
- Employee updates Key Result progress regularly (numeric input or percentage). Progress updates timestamped.
- KPIs (separate from OKRs): HR Admin can define individual KPIs per employee (e.g., 'Resolve 95% of tickets within SLA'). Employee tracks and logs KPI progress.
- OKR/KPI dashboard: employee sees Objective → Key Results → progress bars, colour-coded by confidence level (On Track / At Risk / Off Track).

- Monitor progress: org-wide development completion dashboard; filter by department, plan type, goal type. Export to CSV/Excel.
- Send nudges: HR Admin or manager can send an in-app nudge to employees with overdue goals. Nudge appears as a priority notification.
- Pause / extend plans: HR Admin can pause a plan (freezes due dates) or extend the target date with a reason.

9. PAYROLL & SALARY MANAGEMENT MODULE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Manages payroll data integration, salary record management, payslip distribution, and salary reporting. The platform acts as a payroll data layer that syncs with external payroll engines (ADP, Oracle Payroll, or a custom payroll provider). No payroll processing is performed within the platform — it manages data, triggers, and reporting only.

⚠  The platform does not process payments. It manages employee compensation data and integrates with the organisation's configured payroll provider via scheduled sync or API.

──────────────────────────────────────────────────────────────────────────────
9.1 Employee Salary Records
──────────────────────────────────────────────────────────────────────────────

9.1.1 Salary Profile:
Fields stored per employee:
- Base salary (amount + currency), effective date, salary band/grade.
- Salary history: chronological log of all salary changes (date, old salary, new salary, reason, approved by, changed by). Immutable.
- Pay frequency: monthly, bi-weekly, weekly (configured per org or per employee band).
- Employment type: full-time / part-time / contract. Part-time: FTE ratio stored (e.g., 0.6 FTE) for pro-rated calculations.
- Currency: per employee (supports multi-currency orgs — e.g., LKR for Sri Lanka staff, GBP for UK staff).

9.1.2 Salary Components (Configurable per Org):
- Basic Salary, Housing Allowance, Transport Allowance, Performance Bonus, Overtime Pay, Deductions (EPF/ETF in Sri Lanka; NI/PAYE in UK — applied based on org country setting), Other Allowances.
- Each component configured as: fixed amount, percentage of basic, or variable (entered per pay cycle).
- Component visibility: HR Admin configures which components are visible to employees on their payslip.

──────────────────────────────────────────────────────────────────────────────
9.2 Payslip Management
──────────────────────────────────────────────────────────────────────────────

9.2.1 Payslip Generation:
How it functions:
- Payslip data received from external payroll provider via API sync (scheduled or manually triggered by HR Admin) after payroll is processed.
- Platform parses sync payload → maps fields to employee records → generates formatted payslip PDF per employee.
- Payslips stored securely in encrypted storage. Each payslip linked to: employee_id, pay_period (month/year), generated_at timestamp.
- HR Admin reviews payslip data before 'Publish' — payslips remain in 'Draft' state post-sync.
- HR Admin publishes payslips for all employees at once or by department/batch → each employee receives push notification: 'Your payslip for [Month Year] is ready.'

9.2.2 Employee Payslip Access:
- Employee navigates to Payroll → Payslips tab → sees list of published payslips sorted newest-first.
- Each entry shows: pay period, gross pay, net pay, status (Published / Draft — draft not visible to employee).
- Tapping a payslip opens PDF viewer within the app. Employee can download payslip as PDF.
- History: all payslips retained; employee can access up to 24 months of history by default (configurable).
- Payslip acknowledgement (optional, configurable): employee taps 'Acknowledge' to confirm receipt. Acknowledgement timestamp logged. HR Admin can see acknowledgement status per employee.

──────────────────────────────────────────────────────────────────────────────
9.3 Payroll Sync
──────────────────────────────────────────────────────────────────────────────

9.3.1 Sync Configuration:
- Supported providers: ADP Workforce Now (API), Oracle HCM Payroll (API), and generic CSV import for custom payroll systems.
- Sync schedule: configurable (e.g., monthly on 25th, bi-weekly every Friday). HR Admin can trigger manual sync at any time.
- Sync scope: employee master data pushed to payroll provider; payslip data, deductions, and tax summaries pulled back.

9.3.2 Sync Status & Error Handling:
How it functions:
- Each sync run logged: timestamp, direction (push/pull), records processed, success count, error count, duration.
- Sync errors presented in detail: employee name/ID, field that failed, error message from provider, suggested resolution.
- Critical errors (e.g., employee not found in provider, salary mismatch >10%) trigger alert to HR Admin immediately.
- Non-critical warnings (e.g., missing optional field): queued in warning log; reviewed at next HR Admin session.
- Re-sync specific employees: HR Admin can select individual employees and trigger a targeted re-sync after resolving errors.

──────────────────────────────────────────────────────────────────────────────
9.4 Salary Reports & Export
──────────────────────────────────────────────────────────────────────────────

9.4.1 Salary Summary Report:
How it functions:
- HR Admin navigates to Payroll → Reports → Salary Summary.
- Filter options: Pay Period (month/year), Department, Employment Type, Grade/Band, Currency.
- Report displays: employee name, employee ID, department, grade, basic salary, total allowances, total deductions, net pay, currency, pay period.
- Aggregated totals per department and org-wide.
- 'Export' button → options: CSV, Excel (.xlsx), PDF (formatted report with org letterhead and HR Admin signature block).
- Export audit: every export action logged — { exported_by, export_date, filters_applied, record_count, format }. Immutable. HR Admin and Org Owner can view export history.

9.4.2 Payroll Cost Analysis Report:
- Period-over-period salary cost comparison (e.g., this month vs last month, this year vs last year).
- Breakdown by department: headcount, total payroll cost, average salary, cost as % of total org payroll.
- New hires / leavers impact: shows salary additions and removals that affected total cost in the period.
- Export: CSV, Excel, PDF.

9.4.3 Salary Band Compliance Report:
- Lists employees against their grade/band's configured salary range.
- Flags employees who are below band minimum (red), at minimum (amber), or above band maximum (red).
- Useful for annual compensation review cycles. Export as CSV/Excel.

9.4.4 Individual Salary History Export:
- HR Admin selects an employee → exports their full salary history as PDF or CSV.
- Used for employment reference letters, legal proceedings, or internal audits.
- PDF version includes org letterhead, employee details, and all salary records with change reasons.

9.4.5 Bulk Payslip Export:
- HR Admin can export all payslips for a given pay period as a ZIP of individual PDFs.
- Or export as a merged multi-page PDF (all employee payslips for the month, paginated).
- Useful for compliance archiving. Export logged in audit trail.

- Org Owner can download the full org salary report at any time.
- Salary data is masked for any HR Admin whose access is restricted to specific departments (configurable by Org Owner).

10.1 Natural Language HR Assistant
──────────────────────────────────────────────────────────────────────────────

10.1.1 Intent Parsing & Execution:
How it functions:
- Employee types or voice-inputs a message (e.g., 'Apply annual leave for next Friday').
- NLP intent classifier identifies: intent (apply_leave), entities (leave_type: annual_leave, date: next_friday).
- System resolves date references relative to user's timezone and locale.
- If all required parameters resolved: system presents a confirmation card — 'Leave Request Summary' showing resolved dates, duration, balance impact — with Confirm / Cancel buttons.
- If parameters ambiguous or missing: follow-up prompt shown (e.g., 'What type of leave would you like to apply for?').
- On Confirm: action executed; result confirmation shown inline (e.g., 'Leave request submitted. Pending approval from [Manager Name]').
- All executed actions added to persistent chat history. Tapping a historical action re-opens the relevant record.

Supported Intent Actions:
- Clock In / Clock Out: 'Clock me in as remote' → clock-in flow initiated inline.
- Leave Apply: 'I need a day off on Monday' → guided leave request with balance preview.
- Leave Balance: 'How many annual leave days do I have left?' → balance card response.
- Skill Gap: 'What skills am I missing for my role?' → skill gap summary card.
- Development Plan: 'Show my development goals' → goals list card with progress.
- Payslip: 'Show my last payslip' → payslip card with download button.
- Notifications: 'What do I have pending?' → pending items summary.
- Approvals (team leads): 'Show pending leave approvals' → approval inbox card.

10.1.2 Rich Response Types:
- Info Cards: structured data cards (e.g., attendance summary, leave balance).
- Action Cards: cards with embedded Confirm / Cancel / View Detail buttons.
- Calendar Inline: date picker rendered within chat for date selection.
- Form Inline: short forms (leave reason, correction reason) rendered within chat flow.
- List Cards: scrollable lists for multi-item responses (e.g., list of notifications).

10.1.3 Persistent Chat History:
- All interactions (user messages, assistant responses, action outcomes, notifications acknowledged via chat) stored per employee with timestamp.
- History searchable by keyword. 90-day retention by default (configurable).
- HR Admin cannot read employee chat history (employee privacy). Exception: Org Owner can access audit log of action executions (not the conversational content) for compliance.

HR Admin and Org Owner-facing module for managing the complete employee journey from hire to exit. Covers account creation, status management, role changes, and separation workflows.

──────────────────────────────────────────────────────────────────────────────
12.1 Employee Onboarding (Admin Side)
──────────────────────────────────────────────────────────────────────────────

12.1.1 Single Employee Creation:
How it functions:
- HR Admin navigates to Employees → New Employee.
- Form: First name, Last name, Email (becomes login), Department, Role/Job Title, Grade/Band, Employment Type, Start Date, Reporting Manager (select from org hierarchy), Work Location, Salary (basic + currency + effective date), and optionally: national ID, date of birth.
- On save: employee account created with a temporary password (emailed securely to employee's registered email). Employee status set to 'Active – Onboarding'.
- Onboarding wizard triggered for employee on first login (see Section 2.2).
- HR Admin sees onboarding checklist per new employee: completed steps vs pending, % completion.

12.1.2 Bulk Employee Import:
How it functions:
- HR Admin selects 'Bulk Import' → uploads CSV, Excel (.xlsx), or connects Google Sheets (OAuth).
- Column mapping screen: system auto-detects column headers and maps to platform fields (e.g., 'Full Name' → first_name + last_name split; 'Dept' → department). HR Admin can override auto-mapping.
- Preview screen: first 10 rows shown with mapped values. Validation errors highlighted in red per cell (e.g., invalid email format, unknown department name, missing mandatory field). HR Admin must resolve errors before proceeding.
- Validation options: 'Fix in spreadsheet and re-upload' or 'Fix inline in platform' (editable grid for non-critical errors).
- Import executed: accounts created row by row. Progress bar shown. On completion: success/fail summary (e.g., '142 imported, 3 failed — download error report').
- Rollback option: if import partially fails, HR Admin can rollback entire batch (deletes all accounts created in that batch). Available within 24 hours of import.

──────────────────────────────────────────────────────────────────────────────
12.2 Status Transitions & Role Changes
──────────────────────────────────────────────────────────────────────────────

Employee Status States:
- Active – Onboarding: new employee, completing onboarding wizard.
- Active: fully onboarded and operational.
- On Leave: on approved extended leave (auto-set when leave > 5 days approved; or manually set by HR Admin).
- Suspended: access suspended (e.g., HR investigation). Cannot log in; data preserved.
- Notice Period: resignation or termination notice served; access may be restricted per org policy.
- Inactive / Offboarded: employment ended; account deactivated; data retained per GDPR/PDPA retention policy.

Role & Department Changes:
- HR Admin can update: role, department, grade, reporting manager, work location.
- Each change: effective date set; old values archived in change history. Future-dated changes supported (e.g., 'Role change effective 1st of next month').
- On role change: system updates leave entitlement to new grade's entitlement, required skills matrix updated, development plan targets may be flagged for review.
- Org hierarchy automatically updated when reporting manager changes.

──────────────────────────────────────────────────────────────────────────────
12.3 Offboarding Workflow
──────────────────────────────────────────────────────────────────────────────

How it functions:
- HR Admin initiates offboarding: selects employee → 'Initiate Offboarding' → sets: last working day, reason (resignation / termination / redundancy / end of contract), and exit interview flag.
- Offboarding checklist auto-generated and assigned (configurable per org): equipment return confirmation, access revocation schedule, final payroll trigger, clearance from IT/Finance/HR.
- Each checklist item: assigned to a responsible party (IT Admin, Finance, HR); completion tracked.
- On last working day + 1: employee account access revoked automatically. Data retained per retention policy.
- Final payslip: HR Admin marks final payroll cycle; system triggers payroll sync with 'final pay' flag.
- Leaving certificate / reference letter: can be generated from profile data as a formatted PDF.

18. TALENT DNA MAPPING MODULE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Talent DNA Mapping is the platform's unified intelligence layer that aggregates every data point known about an employee — their skills, certifications, proficiency levels, validation status, development progress, OKR performance, attendance patterns, and role fit — into a single structured talent profile. This profile is then mapped against role requirements, team composition, and organisational needs to surface actionable talent insights automatically.

This module does not replace HR judgment. It eliminates the manual effort of piecing together disconnected data across modules, so HR Admin and Org Owners can make talent decisions based on a complete, live picture of every person in the organisation.

⚠  Talent DNA profiles are built entirely from data that already exists within the platform across other modules. No additional data entry is required from employees or HR Admin beyond what is already collected.

──────────────────────────────────────────────────────────────────────────────
18.1 The Talent DNA Profile
──────────────────────────────────────────────────────────────────────────────

18.1.1 Profile Composition:
Each employee has one Talent DNA Profile. It is a live, auto-updating record composed of the following data layers:

Layer 1 — Skills & Competency Profile:
- All validated skills from Section 7: skill name, category, proficiency level, validation tier (Unvalidated / Manager Validated / HR Verified / Certification Backed).
- Proficiency trend per skill: direction of change over time (improving, static, declining based on proficiency update history).
- Skill coverage score: percentage of Core skills met vs required for current role (0–100%).
- Certification portfolio: active certifications, expiry status, linked skills.

Layer 2 — Development & Growth Profile:
- Active and completed development plans: plan names, completion rates, overdue goals count.
- Goal completion rate: % of development goals completed on time vs total assigned, across all plans.
- Learning velocity: rate of skill acquisition and certification completion over the last 6 / 12 months.
- OKR performance: average Key Result completion % across the last 2 OKR cycles.
- KPI adherence: % of KPI targets met in current and prior period.

Layer 3 — Attendance & Reliability Profile:
- Attendance rate: % of scheduled working days present (rolling 90-day window).
- Punctuality rate: % of days clocked in within grace period (no late arrivals).
- Work mode distribution: % onsite vs remote over the last 90 days.
- Correction request frequency: number of attendance correction requests submitted (high frequency may indicate device issues or reliability concerns).

Layer 4 — Role Fit Score:
How it is calculated:
- System retrieves the role-required skills matrix for the employee's current role (from Section 14.2).
- Compares employee's validated skills and proficiency levels against each required skill.
- Core skills: each met Core skill contributes a weighted score. Core skills below required proficiency contribute a partial score based on the proficiency gap.
- Preferred skills: each met Preferred skill contributes a smaller weighted score.
- Final Role Fit Score expressed as a percentage (0–100%) and a tier label:
  - 90–100% → Excellent Fit
  - 75–89% → Strong Fit
  - 60–74% → Developing Fit
  - 40–59% → Partial Fit
  - Below 40% → Significant Gap
- Score recalculates automatically whenever: a skill is added/updated/validated, a certification is added or expires, or the role-required skills matrix is updated by HR Admin.

Layer 5 — Tenure & Employment Profile:
- Employment start date, tenure in current role, tenure in organisation.
- Grade/band, employment type (full-time / part-time / contract).
- Role change history: number and types of role changes, promotions, lateral moves.
- Time since last salary review (sourced from salary history in Section 9.1).

──────────────────────────────────────────────────────────────────────────────
18.2 Talent Readiness Score
──────────────────────────────────────────────────────────────────────────────

Overview:
The Talent Readiness Score is a composite indicator of how ready an employee is to take on greater responsibility — either through promotion within their current track or through a lateral move into a different role. It is computed from the five profile layers and displayed as a score from 0–100 with a readiness tier.

Score Composition:

Component                        | Weight | Source
---------------------------------|--------|---------------------------------------------
Role Fit Score (current role)    | 30%    | Skills & Certs vs Role Matrix (Section 14.2)
Development Goal Completion Rate | 25%    | Talent Development Module (Section 8)
OKR / KPI Performance            | 20%    | OKRs & KPIs (Section 8.2)
Attendance & Reliability Rate    | 15%    | Attendance Module (Section 4)
Tenure & Growth Trajectory       | 10%    | Employment Profile + Role Change History

Readiness Tiers:
- 80–100 → High Readiness: employee is performing strongly across all dimensions. Suitable for promotion consideration or stretch assignments.
- 65–79 → Ready with Development: employee is performing well but has identified gaps. Suitable for a planned promotion with a targeted development plan.
- 50–64 → Developing: employee is progressing but requires more time or targeted support. Development plan review recommended.
- Below 50 → Needs Attention: significant gaps across one or more dimensions. HR Admin and manager alerted for review and intervention.

⚠  Weights are configurable by Org Owner. An organisation may choose to weight OKR performance higher (e.g., for sales teams) or attendance lower (e.g., for fully remote orgs). Configurable in System Configuration → Talent DNA Settings.

──────────────────────────────────────────────────────────────────────────────
18.3 Succession Candidate Flagging
──────────────────────────────────────────────────────────────────────────────

Overview:
HR Admin and Org Owner can define key roles within the organisation as 'Critical Roles' — positions where vacancy would significantly impact operations. The system automatically identifies and ranks internal candidates for each critical role based on Talent DNA profile data.

How it functions:
- HR Admin navigates to Talent DNA → Succession Planning → 'Mark Role as Critical'.
- For each critical role, HR Admin sets: minimum Readiness Score threshold for candidate consideration (default 60), and optionally: preferred skills weighting overrides for this specific role.
- System scans all active employees org-wide and scores each against the critical role's requirements using the Role Fit calculation from Section 18.1.1 Layer 4 — but applied to the target role's matrix rather than the employee's current role.
- Candidate ranking generated: employees sorted by their fit score for the critical role, with their Readiness Score and key gap summary shown per candidate.
- HR Admin can: tag an employee as a 'Confirmed Succession Candidate' for a critical role, add private HR notes per candidate, or dismiss a candidate (with reason) from the succession list.
- Confirmed candidates automatically get a 'Succession Track' development goal added to their development plan (if HR Admin enables this option) targeting the skill gaps identified for the critical role.
- Succession lists are private — employees are never shown whether they are or are not on a succession list. Only HR Admin and Org Owner can view succession data.

──────────────────────────────────────────────────────────────────────────────
18.4 Internal Mobility Recommendations
──────────────────────────────────────────────────────────────────────────────

Overview:
When a role becomes vacant or is newly created, the platform automatically surfaces internal candidates whose Talent DNA profile is a strong match for that role — before HR considers external recruitment.

How it functions:
- HR Admin creates or marks a role as 'Open' in the Org Hierarchy module (Section 13).
- System automatically runs a talent match scan: scores all active employees against the open role's required skills matrix.
- Internal Mobility Recommendations panel shows: top 5 matched employees, their Role Fit Score for the open role, their current role, their Readiness Score, and key skill gaps to close before they'd be fully ready.
- HR Admin can act on a recommendation: flag the employee for internal interview, assign a bridging development plan targeting the identified gaps, or dismiss the recommendation with a reason.
- Employee is not notified unless HR Admin explicitly triggers an internal opportunity notification — this is a deliberate opt-in to prevent premature or sensitive communications.
- Dismissed recommendations are logged and excluded from future scans for the same role for a configurable period (default 90 days).

──────────────────────────────────────────────────────────────────────────────
18.5 Talent DNA Dashboard (HR Admin & Org Owner)
──────────────────────────────────────────────────────────────────────────────

Org-Level Talent Overview:
- Organisation-wide summary cards: average Role Fit Score, average Readiness Score, % of employees in each readiness tier, employees with High Readiness not yet in a succession plan.
- Department drill-down: click any department to see its talent distribution — individual employee Talent DNA scores, role fit %, readiness tier, and any flagged concerns.
- Skill Coverage Heatmap: org-wide view of which skills are well-covered and which are critically under-represented, mapped across departments (powered by Section 14.3 Skill Gap Dashboard but enriched with readiness data).

Individual Talent DNA Card (HR Admin view per employee):
- All five profile layers displayed in a structured card layout with visual indicators per layer.
- Role Fit Score with breakdown: which Core skills are met, which are below proficiency, which are missing entirely.
- Readiness Score with per-component breakdown and trend arrow (score improving / static / declining vs last quarter).
- Development plan progress linked directly: HR Admin can assign or edit a development plan from within the Talent DNA card.
- Succession flags: whether this employee is a confirmed candidate for any critical role (visible to HR Admin only).
- HR Admin private notes: free-text notes field per employee, visible only to HR Admin and Org Owner. Timestamped per note entry. Not visible to employee.

Talent DNA Reports & Export:
- Talent Readiness Report: all employees with their Readiness Score, tier, and per-component scores. Filterable by department, role, grade, readiness tier. Exportable as CSV / Excel / PDF.
- Role Fit Report: all employees vs their current role's fit score. Surfaces employees who may be misaligned with their current role. Export as CSV / Excel.
- Succession Coverage Report: all critical roles, number of confirmed succession candidates per role, average readiness score of candidates, roles with zero succession coverage flagged in red. Export as PDF (suitable for board-level talent review presentation).
- Internal Mobility Report: open roles, top internal candidates per role, recommended bridging development actions. Export as CSV / Excel.

──────────────────────────────────────────────────────────────────────────────
18.6 Employee View — Career Profile
──────────────────────────────────────────────────────────────────────────────

Employees see a limited, motivating version of their Talent DNA profile under a 'Career Profile' tab. This view is deliberately framed around growth and self-awareness rather than scores and assessments.

What employees CAN see:
- Role Fit Summary: a visual showing which Core and Preferred skills for their current role they have met, and which are gaps — without showing the numerical score. Labelled as 'Your Role Readiness' with a progress-style indicator.
- Skills Strengths: top 5 strongest validated skills highlighted (highest proficiency + highest validation tier).
- Growth Areas: skills identified as gaps for their current role with a 'Start Working On This' action button linking to a development goal or LMS course.
- Development Progress: same as Section 8 development plan view — recontextualised as 'Your Growth Journey'.
- Certifications: active certs with expiry indicators, same as Section 7.2.

What employees CANNOT see:
- Their numerical Readiness Score or Talent Readiness tier.
- Whether they are on any succession candidate list.
- Internal mobility recommendation results.
- HR Admin private notes.
- Other employees' Talent DNA data.

⚠  The Career Profile view is designed to be a constructive self-development tool, not a performance rating display. Score visibility is intentionally restricted to HR Admin and Org Owner to prevent score anxiety and gaming behaviour.

──────────────────────────────────────────────────────────────────────────────

The HR & Talent Development Platform is a cloud-based SaaS system designed to eliminate fragmented HR operations by centralizing employee record management, attendance and leave tracking, organizational hierarchy control, and talent development into a single intelligent platform.

The system is built as a Mobile-First Application with a native iOS and Android app at its core, delivering all HR functionality through a conversational Chat Interface. Full web platform support is also provided, ensuring accessibility across onsite, remote, and hybrid workforces. Initially targeted at organizations in the United Kingdom and Sri Lanka, the architecture is designed for global SaaS scalability.

The system must parse intent, confirm the action, and execute it within the chat thread.
