Platform Overview

Selfvora is a multi-tenant HR SaaS platform built on a Service-Oriented Monolithic Architecture. Every table is tenant-scoped. employees.id is the primary cross-module foreign key. All PKs are UUIDs. The canonical top-level table is tenants. 20 missing HR tables were identified from UI analysis + 19 shared platform tables across 8 services (S1–S8) that power both HR and Work Management.

Key Architectural Decisions

  • Canonical naming: tenants is the single top-level table. Some modules reference it as organisation_id — same FK target.
  • UUID PKs: All tables use uuid primary keys.
  • Cross-module anchor: employees.id is referenced by every module after Infrastructure/Org/Auth.
  • Leave ↔ Payroll bridge: payroll_leave_adjustments is the explicit cross-module bridge table.
  • Biometric circular ref: biometric_events.attendance_record_id is set after processing — back-reference within Attendance module.
  • App-layer FK: payroll_line_items.attendance_record_ref_id is a UUID column with no DB-level FK constraint.
  • All tables integrated: 12 previously-missing tables have been integrated into their module tabs with proper FK references. 8 new-module tables (Notifications, Configuration, Calendar, Reports) remain in the New Module Tables tab. job_families ↔ departments ↔ employees chain is now fully connected. All (ext) FK markers have been resolved to proper FK references.
  • Shared Platform (S1–S8): 19 tables across 8 services powering both HR and Work Management. Covers SSO/auth extensions (S1), tenant billing & feature flags (S2), notification engine (S3), audit compliance (S4), people directory (S6), workflow engine for all approval chains (S7), and real-time presence (S8). File storage (S5) is already handled by file_records.
Foundation — Infrastructure, Org Structure & Auth

The platform foundation: tenants (top-level), users, files, countries reference table, organisational hierarchy (legal entities → offices → departments → job families → job family levels → teams), job titles linked to job families, and the full auth/security stack (roles, permissions, sessions, MFA, audit, consent). 20 tables total.

How the Shared Platform Connects HR & Work Management

  • SSO login (S1): User hits either app → Keycloak OIDC redirect → sso_providers config used → JWT issued → refresh_tokens stored → user_sessions created. Cross-app navigation validates existing JWT — no re-login.
  • Feature gating (S2): Both apps read feature_flags at startup + on change via S8 real-time push. subscription_plans enforce tier limits (max employees, modules enabled). tenant_branding applies custom domain + colors to both UIs.
  • Notification delivery (S3): HR creates notifications row → S3 engine reads notification_templates → renders message → delivers via configured notification_channels (FCM push, Resend email, in-app via S8). escalation_rules trigger on SLA breach.
  • Compliance (S4): retention_policies auto-expire records per GDPR/PDPA rules (market-specific via S2 config). legal_holds override retention for active litigation. compliance_exports generate full data packages on demand.
  • Approval workflows (S7): Leave approval, skill validation, attendance correction, offboarding sign-off (HR) and task approval, timesheet sign-off (Work Management) all use the same engine. One workflow_definitions config per tenant, customisable steps, SLA enforcement, delegation support.
  • Live updates (S8): SignalR hub pushes to per-user channel (approval notifications), per-org broadcast (announcements), per-project channel (Work Management). presence_sessions tracks who's online for presence indicators in both apps.
  • Work Management tables NOT here: Projects, tasks, sprints, boards, time entries, Kanban columns, etc. are Work Management-specific — they'll live in their own module schema and reference users, tenants, employees, teams, departments via cross-module FKs, same pattern as HR modules.
